# ローカルルール追加ガイド

local-UNO の中核目標は「ローカルルールをどんどん追加していく」こと。本書は、コアを改修せずに `rules/` 内でルールを足すための考え方と手順を示す。全体設計は [`spec.md`](./spec.md) を参照。

## 基本方針

- ルール追加は原則 **`rules/` 内で完結**させる。`engine/` は改修しない。
- ルールは「カード固有の効果」と「ゲーム全体の挙動・判定」の2種類に分けて表現する。
- 有効ルールは**順序付きリスト**で持ち、各フックを**先頭から順に**評価する（リデューサ／後勝ち上書き）。
- 有効化リストは**起動時に固定**。順序は**記述順**で決まる。

## ルールの2つの差し込み口

### 1. カード固有の効果

新しい効果カードは、新しい **CardType** を定義して追加する。CardType は色・記号・表示ラベル・画像メタデータ・効果への参照を持つ。

- 画像は CardType のメタデータから自動生成される（未生成分のみ差分生成）。
- カードの一致判定ロジックは CardType に埋め込まない。判定は `can_play` フックで行う。

### 2. ゲーム全体の挙動・判定（フック）

以下のような横断的ルールはエンジンのフックにハンドラを登録して表現する。

- プレイ可能判定（`can_play`）
- プレイ前後・ドロー・ターン終了などのタイミング
- グローバル状態（累積ドロー枚数・強制色・方向）への作用
- 勝敗・得点、山切れ時の挙動
- 手番外アクション（ジャンプイン等）や応答待ちフェーズ（スタック等）：受理可能アクション集合を広げる

## フックの書き方（共通シグネチャ）

すべてのハンドラは共通シグネチャを取る。

```
handler(現在値, state) -> 新しい値
```

- 前のハンドラの出力（現在値）を受け取り、新しい値を返す。
- **単純上書き**: 前値を無視して新しい値を返す。
- **累積**: 前値を使って `現在値 + N` のように返す。

### 例: ドロー2スタック

「ドロー2を出された受け手が、ドローする前にドロー2を重ねられる」場合、累積ドロー枚数のフックで前値に加算する。

```python
def draw2_stack_accumulate(current, state):
    # current は前ルールまでで確定した累積枚数
    return current + 2
```

### 例: ジャンプイン

手番外でも場と完全一致するカードを出せるようにする。`can_play` で、CardType の同一性が成り立つ場合に手番外でも許可を返す。受理可能アクション集合も広げる。

## 順序の考え方

- 標準ルール（`standard`）を常に先頭に固定する。
- 後ろに積むほど上書きが強く効く。
- 相互に影響し合うルール（例:「ドロー2スタック有効」と「ドロー4にドロー2は乗せられない」）は、**記述順**で意図した結果になるよう並べる。

```python
rules = [standard, draw2_stack, jump_in]  # この順で評価される
```

## 追加時のチェックリスト

1. 効果はカード固有か、横断的か？ → CardType 追加か、フック登録かを選ぶ。
2. 新カードを足したなら、CardType にメタデータを持たせたか？（画像は自動生成される）
3. 判定ロジックをカードに埋め込んでいないか？（判定は `can_play`）
4. フックのハンドラは `(現在値, state) -> 新しい値` の形か？累積が必要なら前値を使っているか？
5. 有効化リストの**記述順**は意図どおりか？他ルールとの上書き関係を確認したか？
6. 手番外・応答待ちが必要なら、受理可能アクション集合／フェーズを正しく広げたか？
7. `engine/` を改修せず `rules/` 内で完結したか？
8. ルールの挙動を担保する pytest を書いたか？（`engine/` を純粋に保っているのでルール単体テストが書きやすい）
