# local-UNO 仕様書

二人対戦専用・LAN内専用の、ローカルルール拡張を前提とした UNO 実装。
本書は設計の土台（ベースフレーム）が満たすべき仕様を定める。個々のローカルルールの追加手順は [`rule-authoring.md`](./rule-authoring.md) を参照。

## 1. 目的とスコープ

- ローカルルールを「どんどん追加していく」ことを最優先の設計目標とする。
- 二人対戦専用。多人数向けの抽象化は行わない。
- LAN 内専用。認証・公開ネットワーク対策は行わない。
- ブラウザからアクセスして遊ぶ。プレイヤーはサーバへ接続する2つのブラウザ。

### 土台のスコープ（C 方針）

標準 UNO を一通り動く形で実装する。ただし最初から拡張構造を持たせ、標準ルールを「1つのリファレンス実装のルールセット」として置く。

- 実装するのは標準ルール分のみ。
- 構造としては、カード効果・プレイ可能判定・ターン進行・手番外アクションへの割り込みを後付けできる器を最初から持つ。

## 2. 設計の基本原則

1. **サーバ権威** — ゲーム状態はすべてサーバが保持する。ブラウザは表示と入力送信のみを行う。
2. **エンジンの純粋性・独立性** — ゲームコア（`engine/`）はネットワーク・描画・画像生成を一切知らない。「Action を入れると新しい state が出る」純粋な塊として単体テスト可能に保つ。
3. **ローカルルールは `rules/` 内で完結** — ルール追加でコアを改修しないことを目標とする。

## 3. ゲームエンジンの設計

### 3.1 フックモデル（役割で分離した併用方式）

ローカルルールは2種類の差し込み口で表現する。

- **カード固有の効果** — カード（CardType）が持つ効果として表現する。例: ドロー2/ドロー4の効果。
- **ゲーム全体の挙動・進行・判定** — エンジンのフック（イベント）にハンドラを登録して表現する。例: プレイ可能判定、ジャンプイン、UNO 宣言ペナルティ、勝敗・得点、山切れ時の挙動。

代表的なフック（例、実装時に確定させる）:

- `can_play(card, state)` — プレイ可能判定
- `on_before_play` / `on_after_play` — カードプレイ前後
- `on_draw` — ドロー時
- `on_turn_end` — ターン終了
- グローバル状態（累積ドロー枚数・強制色・方向など）への作用

### 3.2 合成方式（前から順・リデューサ）

- 有効ルールは**順序付きリスト**として持つ。各フックは、そのリストを**先頭から順に**流す**リデューサ／パイプライン方式**で評価する。
- 各ハンドラは共通シグネチャ **`(現在値, state) -> 新しい値`** を取る。
  - 前のハンドラの出力を次のハンドラが受け取る。最後に返った値が採用される（＝後勝ちの上書き）。
  - 単純上書きも累積も同じ仕組みで書ける。例: ドロー2スタックは `現在値 + 2` を返せば累積になる。
- **順序の決定は設定の記述順**とする（priority 数値や依存宣言は用いない）。
  - 標準ルールを常に先頭に固定し、後ろに積むほど上書きが強く効く。
- **有効化リストは起動時に固定**する。ゲームごとの切り替えは行わない。

例（記述順が評価順）:

```python
rules = [standard, draw2_stack, jump_in]
```

### 3.3 手番外アクション・応答待ちの器

標準の厳密ターン制を、より一般的な状態機械の**最小設定**として表現する。

- ゲーム状態は「**今どのプレイヤーの、どんなアクションを受理するか**」を保持する（例: `awaiting: {player, allowed_actions}`）。
- 標準ルール = 受理対象が「手番プレイヤーの play / draw」だけ、という最小の受理集合。
- ローカルルールは受理集合やフェーズを広げることで、以下を後付けできる（**土台では実装しないが、構造上は表現できる**）:
  - **ジャンプイン** — 手番外でも、場と完全一致するカードを出せる。
  - **スタック（ドロー2/4重ね）** — 効果確定前に受け手の応答を待つフェーズを設ける。
  - **UNO 宣言／指摘** — 割り込み・時間窓。

## 4. データモデル

### 4.1 CardType と CardInstance の分離

- **CardType（定義）** — 色・記号・表示ラベル・画像メタデータ・効果への参照を持つ。新カードは新しい CardType を足すだけで追加できる。
- **CardInstance（個体）** — CardType への参照 + 一意 ID を持つ。山札・手札・捨て山にはインスタンスが並ぶ。

この分離により:

- 画像生成は CardType を走査すればよい（§6 と直結）。
- 同じ赤7が複数あってもインスタンス ID で区別でき、クリック／操作対象の特定が確実。
- ジャンプインの「完全一致」は CardType の同一性で自然に表現できる。
- 視界フィルタは個体単位で制御できる（相手手札はインスタンス ID を伏せて枚数のみ、など）。

### 4.2 判定ロジックの置き場所

- CardType には**素の属性のみ**を持たせる。
- 一致判定（色・数字・記号の一致、ワイルドは常に可、など）の**ロジックは `can_play` フックに閉じ込める**。カードに判定ロジックを埋め込まない。

## 5. 通信・実行基盤

- **フレームワーク**: FastAPI（ASGI, uvicorn 実行）。静的ファイル配信と WS エンドポイントを同居させる。
- **通信方式**: WebSocket（双方向・低遅延・サーバからのプッシュ）。
- **状態同期の粒度**: 毎回**フル状態をブロードキャスト**する（差分は送らない）。
  - **クライアントごとに視界フィルタ**を適用する。自分の手札は見える／相手は枚数のみ、など。
  - 差分適用のズレによるバグを避けるため、フル状態送信を基本とする。

## 6. カード画像の生成

- **方式**: Pillow で PNG を生成し、`static/cards/`（配置は実装時に確定）へ保存する。ブラウザは `<img>` で読む。
- **フォント**: 同梱の TTF を1つ bundle し、環境差を消す。
- **メタデータ駆動生成** — 生成器はハードコードの54枚ループにしない。**有効ルールセットが要求する CardType 一覧を走査**して描画する。ルール追加＝カード追加が画像生成まで一気通貫になる。
- **差分生成** — 「必要なカードのうち**未生成のものだけ**描く」。ルール追加時に足りない分だけ自動で増える。
- **全再生成** — `--regenerate` フラグ等で明示的にトリガする。

## 7. ゲームの成立・接続・再接続

- **同時ゲーム数**: サーバは常に**1ゲームだけ**保持する（状態はメモリ上に1つ）。
- **参加方法**: **先着2接続**が自動的にプレイヤー1／2になる。以降は観戦または拒否。名前入力は任意（土台では省略可）。
- **プレイヤー識別・再接続**: **再接続トークン**をブラウザに保存（localStorage / cookie）。
  - リロードや一時切断でも同じトークンで同じプレイヤーとして復帰し、手札を維持する。
  - 復帰時は視界フィルタ済み状態を丸ごと再送して復元する。
- **再戦**: サーバに「リセット／新規対戦」アクションを1つ持たせ、同じ2トークンで盤面を作り直す。サーバ再起動は不要。

## 8. フロントエンド

- **構成**: 素の HTML + バニラ JS（ビルドツール・npm・トランスパイル無し）。サーバが `index.html` / JS / CSS を静的配信する。
- **描画**: 「**状態 → 画面**」を1つの純粋な描画関数に集約する。WebSocket で受け取ったフル状態を丸ごと描き直す。カードは §6 で生成した PNG を `<img>` で並べる。
- **入力**: 「アクションを JSON で送る」だけに限定する。ロジックはサーバ側に寄せ、フロントは薄く保つ。
- **理由**: Python/uv 一本で完結（Node ツールチェーンを持ち込まない）。LAN 内・オフラインでも CDN 不要で動く。

## 9. モジュール構成

```
src/lUNO/
  engine/            # ルール非依存のゲームコア（ネットワーク・描画を一切知らない）
    state.py         #   GameState, PlayerView, awaiting(受理可能アクション)
    cards.py         #   CardType(定義) / CardInstance(個体) / Deck
    actions.py       #   Action の型(play/draw/declare_uno/reset ...)
    hooks.py         #   フック定義とリデューサ実行器（前から順・上書き）
    engine.py        #   Action を受けフックを回し新 state を返す純粋ロジック
  rules/             # ルールセット（プラグイン）
    standard.py      #   標準UNO。can_play / 効果 / 勝敗をフック登録
    __init__.py      #   有効化リスト（起動時固定・記述順）を組み立て
  cards_render/
    generator.py     #   CardType metadata を走査し PNG 差分生成（Pillow）
    assets/          #   同梱フォント等
  server/
    app.py           #   FastAPI: 静的配信 + WS エンドポイント
    session.py       #   1ゲーム状態・2トークン管理・視界フィルタ・ブロードキャスト
  web/               # バニラフロント（静的配信対象）
    index.html / app.js / style.css
  cli.py             # 起動エントリ（画像差分生成→サーバ起動）
```

- **肝**: `engine/` は `server/` も `web/` も Pillow も知らない。エンジンだけを単体テストでき、ローカルルール追加は基本 `rules/` 内で完結する。

## 10. CLI・起動

- **CLI エントリ名**: `luno`。
- **起動フロー**: `uv run luno` で「画像の差分生成 → サーバ起動」を行う。
- 再生成の明示トリガとして `--regenerate` 等のフラグを設ける。

## 11. 開発規約（CLAUDE.md 準拠の要点）

- 言語: Python（>=3.11）、`src/lUNO` レイアウト。
- 環境・依存: uv（`uv.lock` を追跡）。Lint/Format: ruff。テスト: pytest。
- 変更の挙動を担保するテストを必ず書く。特に `engine/` は純粋に保ち単体テストの主対象とする。
- 開発フロー（Issue → ブランチ → 実装+試験 → PR前レビュー → PR → PR後レビュー → マージ）を順守。main への直接コミット禁止。
