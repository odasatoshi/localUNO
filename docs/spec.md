# local-UNO 仕様書

二人対戦専用・LAN内専用の、ローカルルール拡張を前提とした UNO 実装。
本書は設計の土台（ベースフレーム）が満たすべき仕様を定める。個々のローカルルールの追加手順は [`rule-authoring.md`](./rule-authoring.md) を参照。

## 1. 目的とスコープ

- ローカルルールを「どんどん追加していく」ことを最優先の設計目標とする。
- 二人対戦専用。多人数向けの抽象化は行わない。
- LAN 内専用。認証・公開ネットワーク対策は行わない。
- ブラウザからアクセスして遊ぶ。プレイヤーはサーバへ接続する2つのブラウザ。

### 土台のスコープ（C 方針）

標準 UNO を一通り動く形で実装する。ただし最初から拡張構造を持たせ、標準ルールを「1つのリファレンス実装のルールセット」として置く。

- 実装するのは標準ルール分のみ。
- 構造としては、カード効果・プレイ可能判定・ターン進行・手番外アクションへの割り込みを後付けできる器を最初から持つ。

## 2. 設計の基本原則

1. **サーバ権威** — ゲーム状態はすべてサーバが保持する。ブラウザは表示と入力送信のみを行う。
2. **エンジンの純粋性・独立性** — ゲームコア（`engine/`）はネットワーク・描画・画像生成を一切知らない。「Action を入れると新しい state が出る」純粋な塊として単体テスト可能に保つ。非決定要素（シャッフル・配札）は §3.5 の注入 RNG に閉じ込め、純粋性と再現性を保つ。
3. **ローカルルールは `rules/` 内で完結** — ルール追加でコアを改修しないことを目標とする。

## 3. ゲームエンジンの設計

### 3.1 用語

- **GameState** — サーバが持つ**権威状態**の全体（両プレイヤーの手札の中身・山札の順序・捨て山・手番・強制色・方向・累積ドロー枚数・RNG 状態・`awaiting` など）。
- **PlayerView** — 特定プレイヤー向けに GameState を**視界フィルタ**した送信用スナップショット（§5・§6）。クライアントへ送るのは常に PlayerView であり、GameState そのものは送らない。
- **CardType / CardInstance** — §4 参照。
- **Action** — プレイヤー入力（play / draw / choose_color / declare_uno / reset など）。
- **ctx（コンテキスト）** — フックに渡す読み取り専用の評価文脈。少なくとも次を含む: `action`（対象 Action）、`card`（評価対象カード、あれば）、`played_cards`（複数枚出しで出す群、あれば。単数出しは要素1個。`can_stack` 判定と効果適用が参照、house-rules §2）、`hand`（`card` の持ち主の手札）、`top_of_pile`（捨て山トップ）、`current_player`（現手番）、`state`（GameState への参照）。実装で面を追加したら本書へ追記する。

### 3.2 フックの2分類

ローカルルールの差し込み口は次の2種類で表現する。両者を無理に1シグネチャへ統一しない（役割が異なるため明確に分ける）。

**(A) 値リデューサ型フック（value reducer）** — その場で1つの値を先頭から畳み込んで**算出する**（GameState には永続化しない、一時的な判定・計算）。

- シグネチャ: **`handler(現在値, ctx) -> 新しい値`**
- 対象の例:
  - `can_play` → `bool`（プレイ可能判定）
  - `can_stack` → `bool`（複数枚出しで先頭以降のカードを許可するか。シード `False`＝既定は単数のみ。複数枚出しルールが同数字/同記号などで許可を足す、house-rules §2）
  - `score` → `int`（上がり時の得点計算）
- **シード（初期値）は固定定数**（`can_play`・`can_stack` は `False`、`score` は `0`）。毎回この定数から畳み込み直し、結果はその場で消費する（state フィールドへ書き戻さない）。標準ルールが先頭で値を立て、後続ルールが上書きする。
- 値リデューサは**永続フィールドを対象にしない**（次項参照）。累積のように「前の状態を引き継ぐ」ものは値リデューサではなく state トランスフォーマで表現する。

**(B) state トランスフォーマ型フック（state transformer）** — GameState 全体を順に変換する。**永続フィールドの書き換えはすべてこちらが担う**。

- シグネチャ: **`handler(state, ctx) -> state`**
- 対象の例: `on_before_play` / `on_after_play` / `on_draw` / `on_turn_end` / `on_choose_color`（カード効果の適用、手番送り、フェーズ遷移など）
- 「現在値」に相当するのは state 自体。前のハンドラが返した state を次が受け取る。

**永続フィールドの所有権（重要）**: `pending_draw`（累積ドロー枚数）・`forced_color`（強制色）・`direction`（進行方向）・`current_player`（手番）などの**GameState フィールドは GameState が唯一のオーナー**で、書き換えは **state トランスフォーマ型フックのみ**が行う。値リデューサはこれらを扱わない。

- 累積は「現在の state フィールド値を読み、加算して書き戻す」で表現する。例: ドロー2スタックは `state.pending_draw + 2` を書き戻す（シードは常に現在の GameState 値なので、重ねるほど積み上がる）。
- **2人対戦での `direction`**: reverse は一般には実質 skip（手番が自分へ戻る）。`direction` フィールドは保持するが、2人では手番送りの実効は skip と同じとして扱う。
  - ただし**本プロジェクトの有効ルールセットではリバースを無効化する**（[`house-rules.md`](./house-rules.md) §1 が正典、実装は `rules/reverse_off.py`）。すなわち出荷構成ではリバースは skip 化せず「通常の一致で出せる無効果カード」として扱う。上の記述は標準エンジン単体（`rules/standard.py` のみ）の一般論であり、有効ルールセットの挙動は house-rules を参照する。

エンジンの実行器（`engine/hooks.py`）は、フックごとにこの2型を区別して合成する。

### 3.3 合成方式（前から順・記述順）

- 有効ルールは**順序付きリスト**として持つ。各フックは、そのリストを**先頭から順に**評価する。
- 前のハンドラの出力を次のハンドラが受け取る（値リデューサは値、state トランスフォーマは state）。最後に返った結果が採用される（後勝ち）。
- **順序の決定は設定の記述順**とする（priority 数値や依存宣言は用いない）。標準ルールを常に先頭に固定し、後ろに積むほど上書きが強く効く。
- **有効化リストは起動時に固定**する。ゲームごとの切り替えは行わない。

```python
rules = [standard, draw2_stack, jump_in]  # この順で評価される
```

### 3.4 判定フックの合成意味論（OR / AND）

`can_play` のような bool 値リデューサは「後勝ち上書き」だが、**許可を足すルールと制限を課すルールが混在する**ため、初期値と記述順で意味を制御する。

- **初期値は `False`**（何も許可しない）。
- **許可を足すルール**（標準の一致判定、ジャンプインなど）は、条件を満たせば `True` を返す（＝OR 的に許可を追加）。前値が `True` ならそのまま `True` を返す。
- **制限を課すルール**（例:「ドロー4は他に出せる札が無いときのみ」）は、条件に反するとき `False` を返す（＝前がどうであれ却下）。
- したがって**制限ルールは、それが制限したい許可ルールより後ろに置く**。この順序制御が §3.3 の「記述順」の実効的な意味になる。

具体的なハンドラ例は [`rule-authoring.md`](./rule-authoring.md) に示す。

### 3.5 ランダム性（決定的テストのための RNG 注入）

- シャッフル・配札・山切れ再シャッフルは**すべて GameState が保持する RNG（シード付き）を通す**。エンジン内部で `random` のグローバル状態やシステム時刻に触れない。
- ゲーム開始時にシードを与える（サーバが生成、またはテストが固定値を与える）。同じシード＋同じ Action 列 → 同じ GameState、を保証する。
- これにより原則2（純粋性）とテスト再現性を両立する。

### 3.6 手番外アクション・応答待ちの器

標準の厳密ターン制を、より一般的な状態機械の**最小設定**として表現する。

- GameState は「**今どのプレイヤーが、どんなアクションを取れるか**」を保持する。手番外・割り込みを表現できるよう、**プレイヤーごとの受理可能アクションのマップ**とする（`awaiting: {player_id -> [allowed_actions]}`）。単一プレイヤー限定にしない。
- 標準ルール = マップに手番プレイヤーだけが載り、その値が `[play, draw]` のみ、という最小の受理集合。
- ジャンプイン等の手番外アクションは「相手プレイヤーの受理集合に `play`（一致カード限定）を加える」、UNO 指摘等の割り込みは「該当プレイヤーの受理集合に専用アクションを加える」で表現でき、単一 gate による破壊的改修を避けられる。
- ローカルルールは受理集合やフェーズを広げることで、以下を後付けできる（**土台では実装しないが、構造上は表現できる**）:
  - **ジャンプイン** — 手番外でも、場と完全一致するカードを出せる。
  - **スタック（ドロー2/4重ね）** — 効果確定前に受け手の応答を待つ。
  - **UNO 宣言／指摘** — 割り込み・時間窓。

**応答待ちのライフサイクル**（効果適用の途中で入力を待つ仕組み）:

1. state トランスフォーマ型フックが、効果適用の途中で **`awaiting` を設定して state を返す**（＝そこで一旦停止）。
2. エンジンは新しい `awaiting` に合致する Action だけを次に受理する。
3. 対応する Action が来たら、継続フック（例: `on_choose_color`）が残りの効果を適用し、`awaiting` を通常の手番へ戻す。

例（標準のワイルド）: ワイルドを play → 効果フックが「色選択待ち」を `awaiting` に立てて停止 → プレイヤーの `choose_color` Action → 強制色を確定し手番を相手へ送る。スタックやジャンプインも同じライフサイクルの上に載せる。

## 4. データモデル

### 4.1 CardType と CardInstance の分離

- **CardType（定義）** — 色・記号・表示ラベル・画像メタデータ・効果への参照を持つ。新カードは新しい CardType を足すだけで追加できる。
- **CardInstance（個体）** — CardType への参照 + 一意 ID を持つ。山札・手札・捨て山にはインスタンスが並ぶ。

この分離により:

- 画像生成は CardType を走査すればよい（§6 と直結）。
- 同じ赤7が複数あってもインスタンス ID で区別でき、クリック／操作対象の特定が確実。
- ジャンプインの「完全一致」は CardType の同一性で自然に表現できる。
- 視界フィルタは個体単位で制御できる（§5）。

### 4.2 判定ロジックの置き場所

- CardType には**素の属性のみ**を持たせる。
- 一致判定（色・数字・記号の一致、ワイルドは常に可、など）の**ロジックは `can_play` フックに閉じ込める**。カードに判定ロジックを埋め込まない。

## 5. 視界フィルタ（PlayerView の生成）

GameState をそのまま送るとカード情報が漏れる。**ホワイトリスト方式**で、各プレイヤー向けに公開範囲を明示的に決める。

- **全員に公開** — 捨て山のトップカード、現在の強制色・進行方向・累積ドロー枚数、手番プレイヤー、`awaiting`（受理可能アクション）、各プレイヤーの**手札枚数**、山札の**残り枚数**。
- **本人のみ** — 自分の手札の中身（CardInstance の中身）。
- **全員から秘匿** — 山札（draw pile）の**順序と中身**、および相手手札の中身（枚数のみ公開）。RNG のシード・内部状態も送らない。

フィルタは「GameState → PlayerView」の**純関数**として実装し、`engine/state.py` に置く（エンジンの純粋性を保つ）。その関数を呼び出して各クライアントへ配信するのは `server/`（§9）。デフォルトは秘匿とし、公開したい項目だけを明示的に PlayerView へ載せる（うっかり漏れを防ぐ）。

## 6. 通信・実行基盤

- **フレームワーク**: FastAPI（ASGI, uvicorn 実行）。静的ファイル配信と WS エンドポイントを同居させる。
- **通信方式**: WebSocket（双方向・低遅延・サーバからのプッシュ）。
- **状態同期の粒度**: 状態が変わるたびに、各クライアントへ **PlayerView（視界フィルタ済みスナップショット）をフルで**送る（差分は送らない）。差分適用のズレによるバグを避ける。二人・カード数十枚規模では毎回フル送信で十分。

## 7. カード画像の生成

- **方式**: Pillow で PNG を生成し、`src/lUNO/static/cards/`（後述の命名規約で確定）へ保存する。ブラウザは `<img>` で読む。
- **フォント**: 同梱の TTF（`src/lUNO/cards_render/assets/DejaVuSans-Bold.ttf`）を1つ bundle し、環境差を消す。
- **メタデータ駆動生成** — 生成器はハードコードの54枚ループにしない。**有効ルールセットが要求する CardType 一覧を走査**して描画する（実装は `cards_render/generator.py` の `generate_cards()`）。ルール追加＝カード追加が画像生成まで一気通貫になる。
- **差分生成** — 「必要なカードのうち**未生成のものだけ**描く」。同じ見た目（`CardType.image_key`）の札は1画像に畳み、`label`/`effect` だけ違う CardType が増えても画像は増えない（§4.1）。ルール追加時に足りない分だけ自動で増える。
- **全再生成** — `--regenerate` フラグで明示的にトリガする。
- **デザイン**: 公式 UNO の意匠は模倣しないオリジナル。ダーク基調のカード面に色ごとのアクセントで縁取り・中央グリフ・四隅ラベル（右下は180度回転）を描く。skip/reverse/wild は文字でなく独自図形（禁止記号・対向2矢印・4色パネル）で表す。

### 命名・出力規約（§12 の確定事項）

- **出力先**: `src/lUNO/static/cards/`（パッケージ同梱。サーバが静的配信、生成物は git 追跡外＝`.gitignore` 済み）。
- **ファイル名**: `<image_key>.png`。`image_key` は色付き=`<color>_<symbol>`、ワイルド系=`<symbol>`（例: `red_5.png` / `blue_skip.png` / `green_reverse.png` / `wild.png` / `draw4.png`）。
- **画像サイズ**: 300×450 の角丸カード（角の外側は透過 RGBA）。
- フロントの `<img>` はこの命名で参照する（PlayerView のカードから `image_key` を引く）。

## 8. ゲームの成立・接続・再接続

- **同時ゲーム数**: サーバは常に**1ゲームだけ**保持する（GameState はメモリ上に1つ）。
- **参加方法**: **先着2接続**が自動的にプレイヤー1／2になる。名前入力は任意（土台では省略可）。
- **3人目以降**: 土台では**接続拒否**に一本化する（観戦は土台スコープ外。将来ローカルルール/拡張として観戦を足す場合は、両手札を秘匿する専用の視界フィルタを別途定義する）。
- **プレイヤー識別・再接続トークン**:
  - **発行主体はサーバ**。プレイヤーが初回接続した時にサーバがトークンを生成して送り、ブラウザが保存（localStorage / cookie）する。
  - リロードや一時切断でも同じトークンで同じプレイヤーとして復帰し、手札を維持する。復帰時は PlayerView を丸ごと再送して復元する。
  - **多重接続**（同一トークンで複数タブ等）: **後勝ちで接続を置換**する（古い WebSocket は切断）。同時に2画面を正当に使う用途は無い前提。
  - **スロット解放**: プレイヤーが明示的に離脱、またはリセット時にトークンとスロットの対応を作り直せる。
- **再戦**: サーバに「リセット／新規対戦」Action を1つ持たせ、同じ2トークンで盤面を作り直す。サーバ再起動は不要。

## 9. フロントエンド

- **構成**: 素の HTML + バニラ JS（ビルドツール・npm・トランスパイル無し）。サーバが `index.html` / JS / CSS を静的配信する。
- **描画**: 「**PlayerView → 画面**」を1つの純粋な描画関数に集約する。WebSocket で受け取った PlayerView を丸ごと描き直す。カードは §7 で生成した PNG を `<img>` で並べる。
- **入力**: 「Action を JSON で送る」だけに限定する。ロジックはサーバ側に寄せ、フロントは薄く保つ。
- **理由**: Python/uv 一本で完結（Node ツールチェーンを持ち込まない）。LAN 内・オフラインでも CDN 不要で動く。

## 10. モジュール構成

```
src/lUNO/
  engine/            # ルール非依存のゲームコア（ネットワーク・描画を一切知らない）
    state.py         #   GameState, PlayerView, awaiting, 視界フィルタ純関数, RNG
    cards.py         #   CardType(定義) / CardInstance(個体) / Deck
    actions.py       #   Action の型(play/draw/choose_color/declare_uno/reset ...)
    hooks.py         #   フック定義(値リデューサ型 / state トランスフォーマ型)とリデューサ実行器
    engine.py        #   Action を受けフックを回し新 state を返す純粋ロジック
  rules/             # ルールセット（プラグイン）
    standard.py      #   標準UNO。can_play / 効果 / 勝敗をフック登録
    __init__.py      #   有効化リスト（起動時固定・記述順）を組み立て
  cards_render/
    generator.py     #   CardType metadata を走査し PNG 差分生成（Pillow）
    assets/          #   同梱フォント等
  server/
    app.py           #   FastAPI: 静的配信 + WS エンドポイント
    session.py       #   1ゲーム状態・2トークン管理・PlayerView 配信・多重接続処理
  web/               # バニラフロント（静的配信対象）
    index.html / app.js / style.css
  cli.py             # 起動エントリ（画像差分生成→サーバ起動）
```

- **肝**: `engine/` は `server/` も `web/` も Pillow も知らない。エンジンだけを単体テストでき、ローカルルール追加は基本 `rules/` 内で完結する。
- **責務分担**: PlayerView 型と視界フィルタ**純関数**は `engine/`、その呼び出しと WebSocket 配信は `server/`。

## 11. CLI・起動

- **CLI エントリ名**: `luno`。
- **起動フロー**: `uv run luno` で「画像の差分生成 → サーバ起動」を行う。
- 全再生成の明示トリガとして `--regenerate` フラグを設ける。

## 13. 開発規約（CLAUDE.md 準拠の要点）

- 言語: Python（>=3.11）、`src/lUNO` レイアウト。
- 環境・依存: uv（`uv.lock` を追跡）。Lint/Format: ruff。テスト: pytest。
- 変更の挙動を担保するテストを必ず書く。特に `engine/` は純粋（RNG 注入で決定的）に保ち単体テストの主対象とする。
- 開発フロー（Issue → ブランチ → 実装+試験 → PR前レビュー → PR → PR後レビュー → マージ）を順守。main への直接コミット禁止。
